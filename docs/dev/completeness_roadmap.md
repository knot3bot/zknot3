# zknot3 完整度推进路线图（对齐 suif）

## 1. 目标与范围

本文将 `docs/dev/suif.md` 的差距分析转化为可执行路线图，原则如下：

- 先主网门禁，再生态扩展。
- 先协议冻结，再 SDK 扩展，避免实现漂移。
- 所有阶段必须以可量化验收指标收口，不以“代码已完成”作为完成标准。

不在本路线图范围：

- 一次性实现 Sui Framework 全能力等价。
- 在 P0 未稳定前引入高复杂度密码学或合规模块。

---

## 2. 分阶段执行计划

## Phase 0（2-4 周）：主网门禁收口

### 2.0 交付目标

建立“可持续运行 + 可回归验证 + 可观测定位”的主网最小闭环。

### 2.1 工作包

1) Epoch/ValidatorSet 闭环化
- 将 `epoch_bridge` 从可选 fallback 推进为默认主路径。
- 统一 `Node`、`Dashboard`、`RPC/GraphQL` 的 epoch 与验证者视图。
- 建立 epoch 切换回放与一致性测试。

2) 协议冻结 v1（交易 + 签名 + nonce + proof）
- 冻结交易序列化、签名域、nonce/duplicate 语义。
- 输出 golden test vectors（正例、反例、边界值）。
- 建立 SDK 与节点的协议兼容回归测试。

3) DoS 与资源治理补全
- 全入口限流、分页上限、请求体上限、并发上限。
- 高压下保共识链路优先级（降级策略文档化）。
- 对异常流量建立专项回归脚本。

### 2.2 验收指标（Go/No-Go）

- 14d soak：无崩溃、无冻结、无持续内存异常增长。
- Epoch 切换成功率 100%，且可追踪审计。
- 关键接口在压测下满足延迟/错误率门槛（建议：p95、p99、error rate）。
- 重启恢复后状态一致性通过全部回归用例。

---

## Phase 1（3-6 周）：开发者最小闭环

### 1.0 交付目标

做到“外部团队可接入、可验证、可观测”。

### 1.1 工作包

1) SDK 最小闭环
- Zig SDK 从 draft 升为稳定契约版本。
- 增加 1 门主流语言 SDK（建议 TS 或 Rust）。
- 统一错误模型、超时/重试语义、proof 验证接口。

2) 索引/事件端到端可用
- 固化事件模型（类型、字段、版本策略）。
- 打通执行结果 -> 索引入库 -> 查询返回一致性。
- 输出“浏览器/风控/统计”最小查询集。

3) 观测体系产品化
- 指标分层：共识/网络/存储/RPC/proof。
- 引入端到端 trace id。
- 输出标准看板与告警规则。

### 1.2 验收指标

- 新集成方 1 天内可跑通 tx 提交、状态查询、proof 校验。
- 索引查询正确性回归通过，查询性能满足既定 SLO。
- 线上问题定位时间（MTTR）显著下降并可量化。

---

## Phase 2（6-12 周）：能力面扩展（Sui 对标）

### 2.0 交付目标

在稳定基础上扩展开发者原语，提高可组合性与生态可落地性。

### 2.1 工作包

1) Move 能力面增量扩展
- 优先实现高频原语：coin/balance/pay、event、object/transfer/tx_context 等价层。
- 分批推进复杂集合与策略模块，避免一次性扩展过大。

2) 对象语义与并行执行前提增强
- 强化 ownership 与冲突检测模型。
- 增加对象依赖与冲突观测数据，支撑性能调优。

3) 治理与再配置协议化
- 将 validator set 变更能力协议化，而非仅工程配置流程。
- 完成变更审计、回滚、演练机制。

### 2.2 验收指标

- 面向 dApp 的标准场景覆盖率达到预期目标。
- 并行执行收益可量化，冲突行为可解释。
- 治理变更流程具备生产可操作性与演练记录。

---

## 3. 里程碑门禁（统一模板）

每个里程碑必须包含以下门禁项：

- 功能门禁：需求清单全部映射到测试/脚本。
- 性能门禁：压测报告（基线 + 对比 + 结论）。
- 稳定性门禁：soak 报告与异常归因。
- 运维门禁：看板、告警、Runbook、回滚脚本。
- 兼容性门禁：版本策略与协议变更说明。

---

## 4. Issue 模板（可直接复制）

## 模板 A：协议/核心能力类

Title:
`[P0][Protocol] <能力名> v1 冻结与实现`

内容：

- 目标
  - 明确要冻结或实现的协议能力与边界。
- 背景
  - 当前状态、问题与风险。
- 改动点
  - 代码改动清单（模块/接口/数据结构）。
  - 文档改动清单（协议文档、runbook、变更说明）。
- 验收标准
  - 必须通过的功能用例。
  - 必须达成的性能/稳定性指标。
  - 必须补齐的观测项。
- 回滚方案
  - 回滚触发条件。
  - 回滚步骤（含配置、数据、发布策略）。
  - 回滚后验证步骤。
- 风险与缓解
  - 兼容风险、数据风险、性能风险。

---

## 模板 B：索引/数据面类

Title:
`[P1][Indexing] <对象或事件能力> 端到端打通`

内容：

- 目标
  - 明确查询场景与消费者（浏览器/风控/统计）。
- 数据契约
  - 字段定义、版本策略、兼容策略。
- 改动点
  - 生产路径改动（执行 -> 索引）。
  - 消费路径改动（API -> 响应）。
- 验收标准
  - 正确性：执行结果与查询结果一致。
  - 性能：查询延迟与吞吐目标。
  - 可靠性：重启恢复与补数流程验证。
- 回滚方案
  - 索引开关、版本回切、补数策略。

---

## 模板 C：稳定性/运维类

Title:
`[P0][SRE] <模块名> 稳定性门禁补齐`

内容：

- 目标
  - 明确“本次要消灭的故障模式”。
- 观测项
  - 指标、日志、trace、告警阈值。
- 测试计划
  - 压测、故障注入、soak。
- 验收标准
  - 量化门槛（p95/p99、错误率、恢复时间）。
- 回滚方案
  - 灰度/回切步骤与触发机制。

## 5. 建议排期（可按周迭代）

- Week 1-2
  - Phase 0 的协议冻结、epoch 闭环、DoS gate 基线。
- Week 3-4
  - Phase 0 验收与修复、14d soak 启动。
- Week 5-8
  - Phase 1（SDK + 索引 + 观测）并行推进。
- Week 9+
  - Phase 2 按模块增量推进，严格门禁进入。

---

## 6. 执行纪律

- 任何协议变更必须先更新文档与测试向量，再改实现。
- 任何“完成”声明必须附测试/压测/soak证据。
- 任何新增能力必须附观测与回滚路径。
